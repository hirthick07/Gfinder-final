PROJECT OVERVIEW
================================================================================

GFinder is a comprehensive web-based lost and found application designed 
specifically for campus communities (handling 60,000+ students and staff). 
It helps users report lost items, search for found items, and reunite 
belongings with their rightful owners through a modern, secure, and 
user-friendly platform.

The project is a full-stack web application that combines:
- A React-based frontend with TypeScript for type safety
- Supabase Backend-as-a-Service for database, authentication, and storage
- Real-time features for instant messaging and notifications
- AI-powered matching algorithm to connect lost items with found items

DATABASE SCHEMA
================================================================================

TABLES:
-------

1. user_profiles
   - Extends Supabase auth.users table
   - Stores: full_name, phone, bio, avatar_url
   - Preferences: email_notifications, push_notifications
   - Auto-created when user signs up via trigger

2. items
   - Main table for lost/found items
   - Fields: type (lost/found), title, description, category, location, date
   - Status tracking: active/resolved/archived
   - Image storage: image_url pointing to Supabase Storage
   - Performance: Full-text search vector, view counter
   - Foreign key to user_profiles (user_id)

3. messages
   - Direct messaging between users
   - Fields: subject, message, is_read
   - Links: item_id, sender_id, receiver_id
   - Enables item-specific conversations

4. notifications
   - System and user-generated notifications
   - Types: new_message, potential_match, system_alert, item_update
   - Fields: title, message, is_read
   - Links: related_item_id, related_message_id

5. saved_items
   - Many-to-many relationship between users and items
   - Allows users to bookmark/favorite items
   - Composite primary key (user_id, item_id)

6. item_reports
   - Content moderation system
   - Five report reasons with descriptions
   - Status tracking: pending/reviewed/resolved
   - Foreign keys: item_id, reporter_id

DATABASE VIEWS (Virtual Tables):
---------------------------------

1. global_statistics
   - Aggregates: total_items, active_items, resolved_items, total_views
   - Provides dashboard metrics

2. user_statistics
   - Per-user metrics: items_posted, items_resolved, total_item_views
   - Calculates success rates

3. items_with_user_info
   - Joins items with user profile data
   - Simplifies queries needing both datasets

4. potential_item_matches
   - Intelligent matching algorithm
   - Matches lost items with found items based on:
     * Same category
     * Same or nearby location
     * Date within 7 days of each other
   - Provides match scoring

================================================================================
USER WORKFLOW EXAMPLES
================================================================================

SCENARIO 1: REPORTING A LOST ITEM
---------------------------------
1. User creates an account or logs in
2. Navigates to "Report Lost Item" page
3. Fills out form:
   - Title: "Blue iPhone 13 Pro"
   - Description: "Lost near library on 2nd floor"
   - Category: Electronics → Mobile Phone
   - Location: Main Library
   - Date: February 22, 2026
   - Uploads photo of similar phone model
4. Submits form
5. System:
   - Stores item in database
   - Uploads image to Supabase Storage
   - Runs matching algorithm against found items
   - Creates notification if potential matches found
6. User receives confirmation and can view their posted item


SCENARIO 2: SEARCHING FOR FOUND ITEMS
--------------------------------------
1. User opens Browse page
2. Applies filters:
   - Type: Found
   - Category: Electronics
   - Location: Main Library
3. Views search results with images and summaries
4. Clicks on item that matches their lost item
5. Views full details and photos
6. Clicks "Contact Owner" button
7. Sends message explaining why they think it's their item
8. Receives real-time notification when owner responds
9. Arranges meetup to verify and retrieve item
10. Marks item as resolved


SCENARIO 3: REAL-TIME MESSAGING
--------------------------------
1. User A posts a found wallet
2. User B sees it and sends a message
3. System:
   - Creates message in database
   - Sends real-time notification to User A via WebSocket
   - Shows unread badge on User A's notification icon
   - Optionally sends email to User A
4. User A sees notification immediately without refreshing
5. Opens message, reads content, and replies
6. User B receives real-time notification of reply
7. Conversation continues until item is returned


================================================================================
SECURITY FEATURES
================================================================================

1. ROW LEVEL SECURITY (RLS)
   - Users can only view/edit their own profiles
   - Users can only delete their own items
   - Messages require sender or receiver authentication
   - Notifications only visible to intended recipient

2. AUTHENTICATION
   - Encrypted passwords (handled by Supabase)
   - JWT tokens for session management
   - Automatic token refresh
   - Secure HTTP-only cookies

3. DATA VALIDATION
   - Client-side validation with Zod schemas
   - Server-side validation in database constraints
   - File upload restrictions (5MB max, image types only)
   - SQL injection prevention (parameterized queries)

4. PRIVACY
   - Email addresses not publicly visible
   - Phone numbers optional and controlled by user
   - User can control notification preferences
   - Item reports handled privately


================================================================================
PERFORMANCE OPTIMIZATIONS
================================================================================

1. DATABASE INDEXES
   - B-tree indexes on frequently queried columns
   - GIN index for full-text search vectors
   - Composite indexes for complex queries
   - Foreign key indexes for joins

2. CACHING
   - TanStack Query caches API responses (5 min stale time)
   - Automatic background refetching
   - Optimistic UI updates
   - Image caching via browser

3. CODE SPLITTING
   - React lazy loading for routes
   - Dynamic imports for large components
   - Vite's automatic code splitting

4. IMAGE OPTIMIZATION
   - File size limits (5MB max)
   - Supabase automatic image resizing (if configured)

5. REAL-TIME EFFICIENCY
   - WebSocket connection reuse
   - Selective subscriptions (only what user needs)
   - Automatic reconnection on disconnect

GETTING STARTED:
---------------
1. Clone repository from Git
2. Run 'npm install' to install dependencies
3. Create Supabase project at supabase.com

5. Run setup-backend.ps1 to initialize database
6. Run 'npm run dev' to start development server
7. Open http://localhost:5173 in browser

- npm run lint        # Check code quality

1. Run 'npm run build' to create production bundle
2. Deploy 'dist' folder to hosting service (Vercel, Netlify, etc.)
3. Set environment variables on hosting platform
4. Ensure Supabase project is in production mode

DATABASE MIGRATION SYSTEM
================================================================================

The project uses a migration-based approach for database schema:

MIGRATION FILES (executed in order):
-------------------------------------
1. Initial setup migrations (auto-generated IDs)
2. create_user_profiles.sql - User profile system
3. create_messages_system.sql - Messaging functionality
4. add_item_status_and_features.sql - Item management enhancements
5. create_notifications_system.sql - Notification system
6. create_views_and_stats.sql - Analytics and matching
7. create_reports_and_favorites.sql - Moderation and favorites

RUNNING MIGRATIONS:
-------------------
- Automated: .\setup-backend.ps1 (runs all migrations)
- Manual: Copy SQL to Supabase SQL Editor and execute
- Rollback: Not supported (create new migration to undo)

REAL-TIME FEATURES
================================================================================

WEBSOCKET SUBSCRIPTIONS:
-------------------------

1. Notifications Channel
   - Subscribes to user's notification table
   - Listens for INSERT events
   - Shows toast notification immediately
   - Updates notification count badge

2. Messages Channel
   - Subscribes to messages where user is sender/receiver
   - Listens for INSERT and UPDATE events
   - Updates message list in real-time
   - Shows unread message indicators

3. Items Channel (optional)
   - Subscribes to item updates
   - Refreshes item list when new items posted
   - Updates item status changes

================================================================================
COMMON ISSUES & SOLUTIONS
================================================================================

ISSUE: Supabase connection errors
SOLUTION: Check .env file has correct URL and key, verify internet connection

ISSUE: Images not uploading
SOLUTION: Verify storage bucket exists and is public, check file size < 5MB

ISSUE: Real-time not working
SOLUTION: Check Supabase project settings → Realtime is enabled

ISSUE: Database migrations failing
SOLUTION: Run migrations in order, check for syntax errors, verify permissions

ISSUE: Build errors
SOLUTION: Delete node_modules and package-lock.json, run npm install again
